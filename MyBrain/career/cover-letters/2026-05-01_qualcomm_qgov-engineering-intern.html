Dear QGOV Hiring Team,

I'm applying for the Qualcomm Government Technologies Engineering Internship (Summer 2026, San Diego or Boulder). I'm a U.S. Citizen, eligible for export-controlled and government-restricted work without sponsorship — and I'd prefer the San Diego site since I'm a Junior at UC San Diego pursuing a B.S. in Electrical Engineering on the Machine Learning and Controls track, returning to UCSD this fall to finish my final year. Available 11–14 weeks May–September 2026, expected graduation June 2027.

What draws me to QGOV specifically is that government-sponsored R&D — adapting commercial systems to novel applications and environments — is exactly the kind of "make this real-world thing actually work end-to-end" engineering I've been doing at much smaller scale. In ECE 148 last quarter (Winter 2026), I configured a Raspberry Pi 4 stack from scratch for an autonomous vehicle: writing udev rules for USB device permissions, enabling I2C for an OAK-D Lite stereo camera through the DepthAI driver, configuring serial communication with a VESC motor controller, and patching library compatibility issues to make open-source robotics frameworks work with our specific hardware. I then trained a CNN behavior-cloning model with TensorFlow 2.15 and deployed it CPU-only on the Pi — a resource-constrained embedded environment — for full real-time autonomous driving. Across three individual labs (behavior cloning, GPS path following, ROS 2 integration), I got a real sense of how much of an embedded software engineer's work lives at the boundary between "kernel sees the device" and "the system actually behaves correctly."

For my four-person final project (autonomous recycling collector), I owned the Python servo-control script and the Raspberry Pi system integration, while teammates handled YOLOv8 detection, the ROS 2 state machine, and the 3D-printed mechanism. Working across ECE / CompE / Aerospace / Mechanical taught me that delivering an integrated system is mostly about clean interfaces between disciplines — which from what I read is exactly how QGOV ships mission-critical products.

I want to be honest about the gaps relative to the role. The Embedded Software track preferred qualifications include cellular systems, communications systems, and DSP — I haven't worked with cellular protocols (5G / LTE / 802.11) or built DSP pipelines. I also haven't yet written bare-metal microcontroller firmware (my embedded work has been on Linux SBCs) or developed Linux kernel device drivers (I've used drivers heavily, including debugging when udev rules and DepthAI permissions interacted unexpectedly, but I haven't authored kernel-level code). My primary languages are Python, C, and ARM64 Assembly — I'm comfortable in C but haven't done a project specifically in C++. The good news is that the foundations the role asks for — embedded Linux internals, scope-level debugging from coursework, sensor integration, real-time control loops, Git workflow, and ML edge deployment — I've actually done, and I learn embedded tooling fast when there's a working system in front of me.

What's particularly exciting is that QGOV is one of the few places I'd encounter "commercial-to-novel-environment" embedded work where my U.S. citizenship is actually a contributor (not just a non-blocker). Anduril is the comparable opportunity I'm pursuing on this track — both organizations operate where my hands-on RPi-stack experience and the willingness to work under export-control constraints become genuinely useful together.

Thank you for considering my application. I'd be excited to spend a summer doing real government-relevant embedded work in San Diego.
